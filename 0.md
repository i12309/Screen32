Нужно сделать cleanup архитектуры фронта Screen32 после введения generated meta-layer и online/offline runtime.

Контекст:
- generated meta-layer уже есть;
- frontend_runtime.cpp сейчас перегружен и смешивает слишком много ролей;
- offline demo уже должен быть отдельным поведением, а не расползаться по общему runtime;
- README нужно сделать понятным: как все стартует, в каком порядке и кто за что отвечает.

Текущее состояние:
frontend_runtime.cpp сейчас одновременно содержит:
- runtime state,
- config/mode logic,
- transport/client/adapter wiring,
- LVGL UI event bridge,
- service responders,
- page/element state extraction,
- offline demo branching.
Это нужно разложить на отдельные файлы/слои без изменения общей логики.

Цель:
оставить frontend_runtime только orchestration-слоем,
а все остальное вынести в специализированные модули.

========================================
1. НОВАЯ СТРУКТУРА common_app
========================================

Нужно разложить код в src/common_app примерно так:

- frontend_runtime.h / .cpp
  Только orchestration:
  - init
  - tick
  - shutdown/reset если нужно
  - mode selection (online / offline_demo)
  - wiring transport + ScreenClient + EezLvglAdapter + OfflineDemoController

- frontend_service_responder.h / .cpp
  Отдельно:
  - обработка request_device_info
  - request_current_page
  - request_page_state
  - request_element_state
  - сборка и отправка service responses

- frontend_ui_events.h / .cpp
  Отдельно:
  - attach_generated_ui_event_handlers()
  - on_ui_event_cb(...)
  - перевод LVGL events в:
    * button_event
    * input_event
  - переключение поведения online/offline

- page_state_builder.h / .cpp
  Отдельно:
  - сборка PageState / ElementState из generated descriptors + lvgl objects
  - чтение text/value/visible/color-capable state
  - helper-функции чтения текущего UI state

- offline_demo_controller.h / .cpp
  Отдельный класс для demo без связи
  (все demo rules должны быть только здесь)

- frontend_config.h / .cpp
  Конфиг оставить отдельным, не смешивать в runtime

Если часть этих файлов уже есть — использовать их и перераспределить код, а не плодить дубли.

========================================
2. ЧТО ДОЛЖНО ОСТАТЬСЯ В frontend_runtime
========================================

В frontend_runtime должно остаться только:

- загрузка/хранение frontend config
- выбор режима:
  - online
  - offline_demo
- создание/связывание:
  - transport
  - ScreenClient
  - EezLvglAdapter
  - OfflineDemoController
- вызов tick() на правильных слоях

frontend_runtime НЕ должен сам:
- собирать PageState
- читать text/value с lvgl объектов
- вешать LVGL callbacks напрямую
- содержать business/demo navigation rules
- содержать device_info/page_state response logic

========================================
3. OFFLINE DEMO
========================================

Очень важно:

Если в config есть:
- offlineDemo = true

то это означает:
- фронт работает в demo режиме без backend;
- основной runtime не должен содержать demo-логику внутри себя;
- весь demo behavior должен быть только в OfflineDemoController.

Нужно сделать так:

1. OfflineDemoController — единственное место, где живет:
   - порядок страниц
   - next/prev/goto rules
   - button bindings
   - локальная реакция на input/button в demo режиме

2. В основном runtime:
   - только проверка config.offlineDemo
   - если true -> вызвать OfflineDemoController
   - никакой ручной demo logic внутри frontend_runtime/shared_app

3. Поведение старта:
   - если offlineDemo = true:
     * стартуем сразу в offline demo
     * transport/backend не обязателен
   - если offlineDemo = false:
     * online startup
     * показать стартовую страницу из online config
     * отправить hello/device_info
     * ждать команды от backend

========================================
6. README
========================================

Нужно раскрутить в README полную цепочку запуска фронта:

1. Откуда начинается старт:
   - shared_app/app_setup
   - frontend_runtime init
   - config load
   - mode selection

2. Что происходит в online режиме:
   - создать transport
   - создать ScreenClient
   - создать adapter
   - показать первую online страницу
   - отправить hello/device_info
   - ждать backend commands

3. Что происходит в offline_demo режиме:
   - создать UI
   - поднять OfflineDemoController
   - показать первую offline страницу
   - локально обрабатывать кнопки и перелистывание
   - без обязательной связи

4. Как generated meta-layer участвует:
   - page_ids / element_ids / descriptors
   - object map
   - adapter wiring
   - page_state building

README должен отвечать на вопрос:
“с чего начинается фронт, через какие слои проходит, кто за что отвечает”.
