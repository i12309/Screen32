========================================
ОБЩАЯ ЦЕЛЕВАЯ АРХИТЕКТУРА
========================================

Должно быть 4 логических репозитория/проекта:

1. screenLIB - https://github.com/i12309/screenLIB.git
   Это библиотека протокола и runtime-слоев.
   Содержит:
   - core
   - host
   - client
   - adapter API (только интерфейс, не concrete LVGL/EEZ adapter)

2. ScreenUI - https://github.com/i12309/ScreenUI.git
   Это отдельный UI package / design package.
   Содержит:
   - EEZStudio project
   - generated UI
   - LVGL
   - generator tools
   - shared generated meta
   - concrete EezLvglAdapter и frontend object map

3. Screen32 - https://github.com/i12309/Screen32.git
   Это frontend/screen demo/runtime проект.
   Он использует:
   - screenLIB
   - ScreenUI
   Содержит только:
   - integration glue
   - frontend runtime bootstrap
   - config
   - platform-specific startup
   - demo flow
   Не должен форкать ни screenLIB, ни UI package.

4. OLD (или другой backend/business проект) - https://github.com/i12309/machine32.git
   Это backend/host проект.
   Он использует:
   - screenLIB
   - только shared generated meta из ScreenUI
   Не должен тащить LVGL, EEZ generated frontend runtime и concrete UI adapter.

========================================
РОЛИ РЕПОЗИТОРИЕВ
========================================

----------------------------------------
A. screenLIB
----------------------------------------

Назначение:
общая protocol/runtime библиотека.

Оставить в screenLIB:

1. core
   - ITransport
   - FrameCodec
   - ProtoCodec
   - machine.proto / machine.options / machine.pb.*
   - ScreenBridge
   - общие types / ids / contexts, если они не завязаны на конкретный UI package

2. host
   - ScreenConfig
   - ScreenConfigJson
   - ScreenEndpoint
   - ScreenManager
   - ScreenSystem
   - PageRegistry / IPageController
   - host-side transports:
     * UartLink
     * WebSocketServerLink

3. client
   - WebSocketClientLink
   - UartClientLink
   - ScreenClient
   - CommandDispatcher

4. adapter
   ОСТАВИТЬ ТОЛЬКО adapter API:
   - IUiAdapter
   Никакого concrete LVGL/EEZ adapter здесь не должно остаться после реорганизации.

Что убрать/вынести из screenLIB:
- concrete EezLvglAdapter
- UiObjectMap, завязанный на generated UI
- LVGL
- EEZStudio project
- generator scripts по generated UI
Это все должно переехать в ScreenUI.

----------------------------------------
B. ScreenUI
----------------------------------------

Назначение:
единый UI package / design package / generated package.

Должен содержать:

1. eez_project/
   - исходники EEZStudio проекта
   - assets
   - project files
   - все, что является source of truth для UI

2. vendor/
   - LVGL (полная библиотека нужной версии)
   Это frontend-side dependency, используемая UI package.

3. generated/frontend/
   - generated UI code из EEZ/LVGL
   - screens.*
   - ui.*
   - styles/images/fonts и т.п.
   - все, что нужно frontend side для реального UI runtime

4. generated/shared/
   - page_ids.generated.h
   - element_ids.generated.h
   - page_descriptors.generated.h
   - element_descriptors.generated.h
   Это общая generated meta-информация, которую могут использовать и frontend, и backend.

5. generated/frontend_meta/
   - ui_object_map.generated.h
   - ui_object_map.generated.cpp
   - eez_page_meta.generated.cpp
   Это frontend-only generated layer для adapter/runtime.

6. tools/
   - generator scripts
   - generate_ui_meta.py
   - generate_all scripts
   - все build-time codegen tools

7. adapter/lvgl_eez/
   - EezLvglAdapter.h
   - EezLvglAdapter.cpp
   - возможно UiObjectMap wrapper, если не все generated
   Это concrete UI adapter, привязанный к generated UI и LVGL.

Важно:
ScreenUI — это НЕ runtime library “на все”.
Это UI package с shared и frontend-only частями.

Backend должен использовать только:
- generated/shared

Frontend должен использовать:
- generated/frontend
- generated/shared
- generated/frontend_meta
- adapter/lvgl_eez
- vendor/lvgl

----------------------------------------
C. Screen32
----------------------------------------

Назначение:
frontend runtime/demo project.

В Screen32 должно остаться только:

1. common_app/
   - frontend_runtime.*
   - frontend_config.*
   - frontend_platform.h
   - OfflineDemoController (когда будет)
   - другие integration-only frontend слои

2. platform_esp32/
   - platform startup
   - transport creation for esp32
   - local config loading for esp32

3. platform_web/
   - platform startup
   - transport creation for wasm/web
   - local config loading for web

4. demo_web/
   - web build glue
   - frontend_config.json
   - CMake/web startup

5. build glue
   - platformio.ini
   - CMake hooks
   - scripts/*.bat / *.sh для local build

Screen32 НЕ должен содержать:
- копию screenLIB исходников
- копию nanopb runtime
- копию generated UI из ScreenUI, если можно подключить его как dependency/submodule
- копию LVGL, если она уже лежит в ScreenUI

Screen32 — это consumer/integration project.

----------------------------------------
D. OLD / backend project
----------------------------------------

Назначение:
host/business logic.

Должен использовать:
- screenLIB
- generated/shared из ScreenUI

Не должен зависеть от:
- LVGL
- generated/frontend
- EezLvglAdapter
- UiObjectMap.generated.cpp
- frontend runtime

========================================
КОНКРЕТНЫЕ ПРАВИЛА ПЕРЕМЕЩЕНИЯ
========================================

1. Если код зависит от generated EEZ/LVGL UI и lv_obj_t*:
   -> это ScreenUI, не screenLIB.

2. Если код зависит от ITransport / ScreenBridge / machine.proto, но не знает про LVGL:
   -> это screenLIB.

3. Если код отвечает за startup/config/demo/platform glue:
   -> это Screen32.

4. Если код нужен backend/business logic:
   -> это OLD/backend project.

5. Если файл можно использовать и на frontend, и на backend без LVGL:
   -> это shared generated meta в ScreenUI/generated/shared или core/shared в screenLIB.

========================================
ПОДКЛЮЧЕНИЕ МЕЖДУ РЕПОЗИТОРИЯМИ
========================================

Желаемая схема зависимостей:

1. Screen32 использует:
   - screenLIB как submodule/subtree/dependency

3. ScreenUI не должен зависеть от Screen32.
4. ScreenUI может зависеть от screenLIB только если это действительно нужно adapter API.
   Если можно избежать циклических связей — лучше избежать.

Нужно аккуратно проверить и не создать циклическую зависимость между repo.
